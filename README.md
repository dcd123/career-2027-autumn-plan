# 2027 秋招开发岗作战系统

这个仓库不是一份静态计划，而是从 2026-06-22 执行到 2027-08-31 的长期闭环系统。目标是把 Java 后端、AI 应用开发、RAG、Agent、平台开发相关能力沉淀成可检查的代码、笔记、项目 README、压测记录、面试问答和简历素材。

核心原则：

- 每周只有一个主目标。
- 每个任务都必须留下产出物。
- 正常周按 30 小时执行，忙碌周降到 15 小时，崩溃周保留 5 小时不断线。
- 只精细规划最近 4 到 8 周，后续每 4 周滚动更新。
- 项目必须服务秋招简历和面试。

## 如何每周使用

1. 周一打开 `weekly/2026-Wxx.md`，只确认本周唯一主目标。
2. 先做必做任务，选做任务只在必做完成后再做。
3. 每天最多 3 个任务，每个任务记录预计耗时、实际耗时和输出链接。
4. 每周至少完成一次 Git commit，把笔记、代码或 README 更新提交进去。
5. 周末按 `review/weekly-review-template.md` 完成复盘，并调整下一周计划。

每周闭环固定为：

- 输入：看课、文档、书籍、文章。
- 练习：刷题、实验、编码。
- 输出：笔记、代码、README、测试记录。
- 验证：测试、复述、面试题、压测结果。
- 复盘：总结卡点，砍掉低价值任务，确定下周最小目标。

## 如何复盘

复盘只回答能改变行动的问题：

- 本周完成了什么，有哪些链接或提交记录？
- 哪个任务最卡，卡在概念、环境、编码还是表达？
- 哪个任务下周要砍掉或降级？
- 哪些内容能写进简历或项目介绍？
- 下周最小可执行目标是什么？

周复盘使用 `review/weekly-review-template.md`，月度复盘使用 `review/monthly-review-template.md`。

## 如何在低状态周继续执行

低状态周不追求补齐原计划，只执行 `review/failure-recovery.md` 的 5 小时版本：

- 3 道算法题。
- 1 篇短笔记。
- 1 次项目 commit。
- 1 次周复盘。

这套动作的目的不是赶进度，而是避免完全中断。恢复后从当前周继续，不补旧账。

## 如何让 Codex 每周辅助检查

每周开始时可以让 Codex 做三件事：

1. 检查本周计划是否包含必需字段。
2. 根据上周复盘，更新下一周任务，保持一个主目标。
3. 审查项目代码和 README，指出能否支撑面试讲述。

检查命令：

```bash
python scripts/check_week.py
```

建议给 Codex 的提示：

```text
请检查 weekly/2026-Wxx.md 是否符合仓库规则，并根据上周复盘把下一周计划降级到可执行版本。
```

## 如何把项目内容转化成简历

每个项目都要同步维护：

- 项目 README：背景、技术栈、核心功能、接口、部署方式。
- 关键代码模块：能展示设计取舍和工程质量。
- 测试或压测记录：用数据证明效果。
- 面试问答：写在 `interview/project-interview-qa.md`。
- 简历 bullet：写在 `resume/resume-project-bullets.md`。

简历 bullet 模板：

```text
基于 Spring Boot + MySQL + Redis 实现 xxx 系统，负责 xxx 模块；通过 xxx 方案解决 xxx 问题，使 xxx 指标达到 xxx。
```

不要只写“使用了 Redis/MQ/RAG/Agent”，要写清楚场景、问题、方案和可验证结果。
